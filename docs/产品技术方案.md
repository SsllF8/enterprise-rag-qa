## 一、项目概述

### 1.1 背景

随着公司业务规模扩大，当前智能客服系统（V1.0）面临以下痛点：
- 客服机器人准确率仅78%，用户满意度下降
- 单日最高并发量已触及系统上限（5000 QPS）
- 多轮对话能力不足，复杂问题需转人工比例高达35%
- 知识库更新依赖人工维护，平均延迟48小时

### 1.2 目标

Pro 版核心目标：
- 问答准确率提升至 92% 以上
- 系统支持 20000 QPS
- 多轮对话完成率提升至 80%
- 知识库更新延迟降低至 1 小时以内

## 二、技术架构

### 2.1 整体架构

系统采用微服务架构，分为四层：

1. **接入层**：Nginx 负载均衡 + API Gateway（Kong）
2. **业务层**：对话管理服务、意图识别服务、知识检索服务、工单服务
3. **AI 层**：RAG 检索引擎、LLM 推理服务、对话状态管理
4. **数据层**：PostgreSQL（业务数据）、Elasticsearch（全文检索）、ChromaDB（向量检索）、Redis（缓存）

### 2.2 RAG 引擎设计

这是 Pro 版的核心升级点：

**检索策略**：混合检索（Hybrid Retrieval）
- BM25 关键词检索 + 向量语义检索，加权融合
- 关键词检索权重 0.3，语义检索权重 0.7
- Rerank 模型使用 BGE-Reranker-large，对 Top-20 结果重排序

**分块策略**：
- 文档按语义段落分块，块大小 512 tokens，重叠 64 tokens
- 表格数据单独提取，保留结构化格式
- 长文档采用层级索引（父文档 + 子文档）

**LLM 配置**：
- 主力模型：DeepSeek-V3（通用问答）
- 快速响应模型：DeepSeek-V2-Lite（简单查询）
- 推理采用流式输出，首 Token 延迟 < 500ms
- Token 上限：单次对话 4096 tokens

### 2.3 向量数据库选型

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| ChromaDB | 轻量、易部署 | 大规模性能一般 | 开发/测试环境 |
| Milvus | 高性能、支持百亿向量 | 运维复杂 | 生产环境 |
| Pinecone | 全托管、免运维 | 成本高、数据在海外 | 不采用 |

生产环境选择 Milvus 集群模式，预估向量规模 5000 万条。

## 三、核心功能模块

### 3.1 多轮对话管理

采用状态机模型管理对话流程：
- 意图识别 → 槽位填充 → 知识检索 → 答案生成 → 确认追问
- 支持打断、换话题、回溯等操作
- 对话上下文保留最近 10 轮

### 3.2 知识库管理

- 支持导入格式：PDF、Word、Excel、网页（URL自动抓取）
- 知识库分类：产品文档、FAQ、政策制度、历史工单
- 支持增量更新，无需全量重建索引
- 知识质量评分：定期检测知识库覆盖率、重复率、过时内容

### 3.3 人工转接

触发条件：
- 用户连续 2 次表示"不满意"
- 系统置信度低于 60%
- 检测到紧急/投诉类意图
- 用户主动要求转人工

转接信息自动携带完整对话历史和用户画像。

### 3.4 数据分析看板

核心指标：
- 问答准确率、用户满意度、平均响应时间
- Top-10 高频问题、未解决问题列表
- 人工转接率、首次解决率
- 按渠道/时间段的流量分布
